# InsureTech TypeScript SDK Test Suite

Comprehensive test suite for the InsureTech TypeScript SDK with 80%+ coverage target.

## Test Structure

```
tests/
├── setup.ts                          # Global test setup with MSW
├── helpers/
│   ├── mock-server.ts               # MSW mock server with API handlers
│   ├── test-data.ts                 # Test fixtures and mock data
│   └── test-utils.ts                # Utility functions for testing
├── unit/
│   ├── client.test.ts               # Client initialization tests
│   ├── types.test.ts                # Type validation tests
│   └── exports.test.ts              # Module exports tests
├── integration/
│   ├── auth/
│   │   ├── registration.test.ts     # User registration flows
│   │   ├── login.test.ts            # Login flows (mobile & email)
│   │   ├── otp.test.ts              # OTP management
│   │   ├── session.test.ts          # Session management
│   │   ├── password.test.ts         # Password operations
│   │   └── token.test.ts            # Token validation & refresh
│   ├── policy/
│   │   └── policy.test.ts           # Policy CRUD operations
│   ├── claim/
│   │   └── claim.test.ts            # Claim submission & processing
│   └── product/
│       └── product.test.ts          # Product operations
└── e2e/
    ├── complete-flow.test.ts        # End-to-end user journeys
    └── error-handling.test.ts       # Error scenarios
```

## Running Tests

```bash
# Run all tests
npm test

# Run tests in watch mode
npm run test:watch

# Run tests with coverage
npm run test:coverage

# Run specific test file
npm test -- tests/integration/auth/login.test.ts

# Run tests matching pattern
npm test -- --grep "Auth - Login"
```

## Test Categories

### Unit Tests (tests/unit/)
Tests for individual components without external dependencies:
- Client configuration and initialization
- Type definitions and interfaces
- Module exports and public API

### Integration Tests (tests/integration/)
Tests for service methods with mocked API responses:

#### Authentication (auth/)
- **registration.test.ts**: Mobile & email registration flows
- **login.test.ts**: Login with mobile/email, logout
- **otp.test.ts**: OTP sending, verification, expiration
- **session.test.ts**: Session CRUD, revocation
- **password.test.ts**: Change password, reset flows
- **token.test.ts**: Token refresh, validation, CSRF

#### Policy (policy/)
- **policy.test.ts**: Create, read, update, cancel, renew, issue policies

#### Claim (claim/)
- **claim.test.ts**: Submit, approve, reject, settle claims

#### Product (product/)
- **product.test.ts**: List products, calculate premiums, search

### E2E Tests (tests/e2e/)
Complete user journey tests:
- **complete-flow.test.ts**: Full registration → purchase → claim flow
- **error-handling.test.ts**: Error scenarios and edge cases

## Mock Server (MSW)

The test suite uses [Mock Service Worker (MSW)](https://mswjs.io/) to intercept HTTP requests and return mock responses.

### Mock Handlers
All API endpoints are mocked in `tests/helpers/mock-server.ts`:
- Auth endpoints (register, login, OTP, sessions, tokens)
- Policy endpoints (CRUD operations)
- Claim endpoints (submission, processing)
- Product endpoints (listing, premium calculation)

### Customizing Mocks
To add or modify mock handlers:

```typescript
// tests/helpers/mock-server.ts
import { http, HttpResponse } from 'msw';

const handlers = [
  http.post('http://localhost:3000/v1/your-endpoint', async ({ request }) => {
    const body = await request.json();
    return HttpResponse.json({ success: true }, { status: 200 });
  }),
];
```

## Writing New Tests

### 1. Unit Test Example

```typescript
import { describe, it, expect } from 'vitest';

describe('MyFeature', () => {
  it('should do something', () => {
    expect(true).toBe(true);
  });
});
```

### 2. Integration Test Example

```typescript
import { describe, it, expect } from 'vitest';
import { myServiceMethod } from '../../src/sdk.gen';
import { createTestClient } from '../helpers/test-utils';

describe('My Service', () => {
  it('should call API successfully', async () => {
    const client = createTestClient();
    const response = await myServiceMethod({
      client,
      body: { data: 'test' },
    });
    
    expect(response.data).toBeDefined();
  });
});
```

### 3. E2E Test Example

```typescript
import { describe, it, expect } from 'vitest';
import { authServiceLogin, policyServiceCreatePolicy } from '../../src/sdk.gen';
import { createTestClient } from '../helpers/test-utils';

describe('E2E - User Flow', () => {
  it('should complete full flow', async () => {
    const client = createTestClient();
    
    // Step 1: Login
    const loginResponse = await authServiceLogin({ client, body: {...} });
    expect(loginResponse.data?.access_token).toBeDefined();
    
    // Step 2: Create policy
    const policyResponse = await policyServiceCreatePolicy({ client, body: {...} });
    expect(policyResponse.data?.policy_id).toBeDefined();
  });
});
```

## Troubleshooting

### Tests failing with "Cannot find module"
```bash
npm install
npm run build
```

### MSW not intercepting requests
Check that `tests/setup.ts` is configured in `vitest.config.ts`:
```typescript
export default defineConfig({
  test: {
    setupFiles: ['./tests/setup.ts'],
  },
});
```

### Coverage not excluding generated files
Update `vitest.config.ts`:
```typescript
coverage: {
  exclude: [
    'src/sdk.gen.ts',
    'src/types.gen.ts',
    // ...
  ],
}
```

## Performance

All tests should complete in < 30 seconds:
- Unit tests: < 1 second
- Integration tests: < 10 seconds
- E2E tests: < 20 seconds

If tests are slow:
1. Check for unnecessary `await` calls
2. Reduce test data size
3. Use `vi.mock()` for heavy dependencies

## Best Practices

1. **Test Isolation**: Each test should be independent
2. **Clear Names**: Use descriptive test names
3. **Arrange-Act-Assert**: Follow AAA pattern
4. **Mock External Calls**: Use MSW for all HTTP requests
5. **Avoid Hardcoding**: Use test fixtures from `test-data.ts`
6. **Test Edge Cases**: Include error scenarios
7. **Keep Tests Fast**: Avoid unnecessary delays
8. **Maintain Coverage**: Add tests for new features
